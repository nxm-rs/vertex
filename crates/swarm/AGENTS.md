# AGENTS: crates/swarm/

Swarm domain layer. Defines what a Swarm node is, the network spec, identity, peer records, bandwidth accounting, peer manager, topology, localstore and storer config, redistribution config, and the RPC service surface. The libp2p-aware composition lives in `swarm/node` and `swarm/topology`; the rest is protocol-agnostic.

Root-level rules in `/AGENTS.md` apply here too. The notes below are the area-specific overlay. The wire-protocol crates under `crates/swarm/net/` have their own `AGENTS.md`.

## Sub-area map

- `primitives`, `forks`, `spec`, `identity`: pure data and configuration. `nectar-primitives` is the canonical source for chunk and address types.
- `peers/peer`: the `SwarmPeer` record with the EIP-191 handshake signature (distinct from postage EIP-191 signing, which lives upstream in `nectar-postage`). See the follow-up tracked at `nxm-rs/vertex` for extracting the sign-data primitive to nectar.
- `peers/peer-manager`, `peers/peer-score`: lifecycle, persistence, scoring. The manager is the authoritative peer hub and holds the ENTIRE known peer set in memory (one `DashMap`; the `ProximityIndex` is a pure bin-membership index with a per-bin admission cap). Persistence is an opt-in identity-only snapshot (`PeerSnapshot` via the `PeerSnapshotStore` trait from `vertex-net-peer-store`, present only when the node database is enabled via `--db.persist`/`--db.path`; the default is fully in-memory), loaded once at startup and written periodically by `PeerManager::tick` plus a final write on shutdown; reputation, bans, and dial backoff are runtime-only and never survive a restart. The manager owns no timers: a thin `spawn_peer_manager_task` driver is spawned from the node launch path. Subsystems change a peer's score only through `PeerManager::report_peer` (the `PeerReporter` trait from `api`), and the manager broadcasts `PeerLifecycleEvent`s that topology consumes to execute disconnects and bans. `peer-score` is policy-only: `record_event` returns a `ScoreOutcome`, it never executes actions and has no callback hooks. The manager also owns IP association tracking for identity-cycling detection: topology passes the connection's remote IP into `on_peer_connected`, sightings are grouped per IPv4 address and per IPv6 /64, cap crossings are reported through `report_peer` (score-based, never a direct ban), and local-subnet/trusted peers are exempt (see `docs/networking/peer-management.md`). The whole `peers/*` stack is wasm-bound: it must keep building for `wasm32-unknown-unknown` (the `wasm` CI job enforces this), wall clocks go through `web-time`, tokio stays at the `sync` feature in the manager, and test-only deps (`arbitrary`) stay behind the `test-utils` feature. See `docs/agents/wasm.md`.
- `accounting/{core,pricing,pseudosettle,swap,chequebook}`: per-peer balances in Accounting Units (AU) and the settlement providers. `Accounting` implements `PeerAffordability` and the accounting and settlement services take an optional `PeerReporter` (both from `api`) so violations feed peer scoring; the node layer does the wiring.
- `api`: the trait chain `SwarmPrimitives` to `SwarmNetworkTypes` to `SwarmClientTypes` to `SwarmStorerTypes`. Strictly libp2p-free with the documented `Multiaddr` exception.
- `client-protocol`: the command and event contract for the client behaviour (`ClientCommand`, `ClientEvent`, `FailureKind`, the `RetrievalResponseTx`/`PushResponseTx` response channels, `ChunkTransferError`, `RetrievalResult`, and the extracted `PseudosettleEvent`/`SwapEvent`). Sits below both `node` and the settlement crates so neither depends up on the other: `node` re-exports every item from its original paths, and `accounting/{pseudosettle,swap}` import the contract directly. Pure data, no `NetworkBehaviour` or `Swarm`; uses libp2p only for `PeerId` in the per-connection command/event variants.
- `builder`: layered builders that produce `BuiltBootnode`, `BuiltClient`, `BuiltStorer`.
- `node`: composes the libp2p `NetworkBehaviour` and exposes `BootNode`, `ClientNode`, `StorerNode`. This is where libp2p shows up. Also owns `PeerSelector`: retrieval and pushsync candidate selection ordered by peer score and affordability on top of proximity. The builder wires it from the topology handle (scores via the peer manager) and the bandwidth accounting (affordability, per-peer chunk price, best-effort settlement trigger). Also owns `SelfThrottle`: the outbound self-throttle (`throttle.rs`) that paces our own retrieval and pushsync requests under each peer's pseudosettle allowance before `ClientHandle::retrieve_chunk`/`push_chunk` dispatches. One throttle is shared by both protocols; `SelfThrottle::new` takes the client accounting object plus a `&DefaultBandwidthConfig` and derives everything from them (the allowance signal from `accounting.bandwidth()`, the pricer from `accounting.pricing()`, the refresh rate and margin percent off the config), so its allowance signal is the same `PeerAffordability` the selector reads (built once in accounting, never hard-coded), one token is one AU so the bucket refills at the `refresh_rate` AU per second the remote forgives our debt and a request costs the exact per-chunk proximity price the remote meters (`SwarmPricing::peer_price`, the same figure accounting debits, for both protocols; retrieval has no separate size term), and the per-peer bucket is resized on every allowance change and cleared on disconnect. The bucket capacity is `PeerAffordability::allowance_to_payment_threshold` (the live headroom toward the settlement trigger, which already includes the credit earned by serving that peer, since our balance rises when we provide to them) scaled by `--bandwidth.throttle-allowance-percent` (default 85), so a post-flap burst stays below the swap trigger with a margin to spare; pacing is bilateral and per-peer with no cross-peer pooling. Tripping it increments `pushsync_self_throttled_total{peer_overlay}` / `retrieval_self_throttled_total{peer_overlay}`. The client handler (`protocol/handler.rs`) carries a named per-protocol outbound deadline for each chunk-transfer operation: `Config::retrieval_timeout` and `Config::pushsync_timeout` (both default 30s) thread into the retrieval and pushsync `SubstreamProtocol::with_timeout`, while pricing, pseudosettle, and swap stay on the shared `Config::timeout`. The split is deliberate (the retrieval/pushsync liveness invariant against a withholding peer is bounded by these two and nothing else), so do not collapse the three fields. When the deadline fires the substream upgrade resolves the caller's response with the typed `ChunkTransferError::TimedOut` (still `FailureKind::Protocol` for scoring, so the existing -2.0 path is unchanged) and increments the bounded, label-free counters `swarm.client.retrieval_timeouts_total` / `swarm.client.pushsync_timeouts_total`; per-peer detail lives in scoring and the structured debug log, never a `peer_overlay` metric label.
- `topology`: libp2p behaviour for peer discovery, kademlia routing, reachability tracking. Also owns the deterministic readiness surface: `TopologyHandle::readiness` snapshots exact connected counts, depth, and per-bin shortfalls from the routing table and peer manager (never from metrics), and `wait_until` plus its shorthands (`wait_until_routable`, `wait_until_depth`, `wait_until_saturated`, `wait_until_ready`) await readiness conditions event-driven via the `TopologyEvent` broadcast. The neighborhood readiness gate for pull-syncing is part of this surface: the routing table keeps a stability clock (re-anchored on every table mutation; cleared on a saturation dip, restarted on a depth change), `ReadinessSnapshot::is_neighborhood_ready` requires the neighborhood to have stayed saturated at an unchanged depth for the `KademliaConfig` stability window, and `wait_until_neighborhood_ready` awaits it with a timer armed alongside the event subscription (time alone can complete the window).
- `localstore`: the `SwarmLocalStore` client cache (`ChunkStore`, a thin wrapper over the byte-bounded LRU in `vertex-store`) plus its config. The cache value is a `CachedChunk` (a chunk plus an *optional* stamp): a content chunk retrieved from a storer arrives stampless and is cached by address and served indefinitely (content is immutable), while a single-owner chunk always carries the stamp whose signed timestamp orders its versions and is served only while within `soc_cache_ttl`, with a newer SOC winning on insert. The retrieval path never caches a SOC at all, since a stampless SOC has no version signal. The persisting storer reserve implements the same `SwarmLocalStore` trait separately and is tracked apart from the cache-only client.
- `storer`, `redistribution`: storer-side configuration and the chunk-store/reserve backend (`DbChunkStore`, `Reserve`).
- `rpc`: tonic-generated gRPC services and a `GrpcServiceProvider` trait.
- `test-utils`: `MockIdentity`, `MockTopology`, and fixtures.

## libp2p dependency

- libp2p-free: `primitives`, `forks`, `spec`, `identity`, `api` (apart from the `Multiaddr` re-export), `builder` (almost), `accounting/*`, `localstore`, `redistribution`, `storer`, `rpc`, `test-utils`, `peers/peer-manager`, `peers/peer-score`.
- libp2p-aware: `peers/peer` (uses `Multiaddr`, `PeerId`), `client-protocol` (`PeerId` only, no `NetworkBehaviour`), `node`, `topology`.

When in doubt: if you need a `NetworkBehaviour` or a `Swarm`, you are in `swarm/node`, `swarm/topology`, or a future composite-behaviour crate for a node type. If you need only `Multiaddr` or `PeerId` you can be elsewhere, but justify it.

### Client and storer protocols

The protocol crates under `crates/swarm/net/` are codec-and-upgrade crates, not `NetworkBehaviour`s. The client-side protocols (pricing, retrieval, pushsync, pseudosettle, and swap behind its feature) are multiplexed by a single hand-rolled `ClientBehaviour`/`ClientHandler` over headered substreams in `swarm/node` (`protocol/`). There is no `#[derive(NetworkBehaviour)]` composition of these protocols, and the handler's substream multiplexing, back-pressure, and per-protocol timeouts are load-bearing; do not reshape them into derived sub-behaviours.

The shared command and event contract between that behaviour and its settlement services lives in `vertex-swarm-client-protocol`, below both `node` and `accounting/*`, so the settlement crates depend down on the contract rather than up on `node`.

Plan of record: composite behaviour crates per node type, mirroring `topology`. `vertex-swarm-client-behaviour` extracts the hand-rolled `ClientBehaviour`/`ClientHandler`/upgrade/forward/`StorerCapability` as-is (still one `NetworkBehaviour`, never reshaped into derived sub-behaviours), and `vertex-swarm-storer-behaviour` is a `#[derive(NetworkBehaviour)]` composite of the client behaviour plus a `PullsyncBehaviour`. Both live in `crates/swarm/` (libp2p-aware tier), keep `crates/swarm/net/` one-protocol-per-crate, depend down on `vertex-swarm-client-protocol` and `api` traits (never up on `node` or concrete accounting), and let `swarm/node` select the composite per node type. Until the extraction lands, the client protocols are still wired in `swarm/node` (`protocol/`) as described above. See `docs/design/composite-behaviours.md` for the layering, the no-cycle argument, and the sequenced E1-E5 plan.

## Nectar boundary

Primitives and layer-2 constructs (chunks, addresses, BMT, manifests, feeds, postage, erasure) live in `nectar` (https://github.com/nxm-rs/nectar), not in vertex. Before adding a new type to `primitives`, `spec`, or any sibling crate, check the Repo split section in `/AGENTS.md`. If the new type is non-node-specific, file the PR against `nxm-rs/nectar` and consume it here through `vertex-swarm-primitives` once merged. The workspace pins all nectar deps to the same git rev (`/Cargo.toml`).

## Dos

- Treat `vertex-swarm-api` as the trait surface. New domain capabilities go in an api trait first, then in a concrete implementation crate.
- Use `nectar-primitives` re-exports (`OverlayAddress`, `Bin`, `ProximityOrder`, `Nonce`, `Timestamp`) so the canonical types stay consistent across the workspace.
- Keep the three proximity types distinct; do not collapse them to `u8` (the bee `po: u8` habit). `ProximityOrder` is the metric between two addresses (rank by closeness to a target); `Bin` is a peer's slot index in the local table (keys per-bin storage/iteration); `NeighborhoodDepth` is the boundary bin. Bridges: `Bin::from(po)` is the only `ProximityOrder -> Bin` conversion; `NeighborhoodDepth::{new, bin, contains}` are the only ways in/out of a depth. `NeighborhoodDepth` is intentionally NOT comparable with `Bin` - write `depth.contains(bin)`, never `bin >= depth`. Enumerate bins only via `all_bins`/`balanced_bins(depth)`/`neighborhood_bins(depth, max)`; extract `.get()` only at edges (logs, metrics, wire). Full rationale: `vertex-swarm-primitives` crate docs.
- When `vertex-swarm-primitives` would host a new type that has no node dependency, push it upstream to `nectar` instead and re-export it here.
- Error types are `thiserror` enums with `strum::IntoStaticStr` so they emit a `reason` label cleanly.
- Use accounting units (AU) in bandwidth code. Never mix bytes, BZZ, and AU in the same struct field. The typed `Au` newtype in `vertex-swarm-api` (re-exported as `vertex_swarm_api::Au`) carries every AU-denominated balance, price, threshold, reserve, allowance, and payment; it is signed (positive = peer owes us). Build non-negative amounts with `Au::from_amount`, balances with `Au::new`. Do the proximity price and the `rate * elapsed` allowance through `Au::checked_scale` so the two overflow-prone multiplications stay audited. The only sanctioned AU boundary conversions are the bytes-to-AU pricing step, the standard `TryFrom` impls between `Au` and `U256` on the newtype (both directions are fallible because `Au` is a signed `i64` and `U256` is unsigned and far larger), and the `U256` to/from AU helpers in the pseudosettle service; do not reintroduce raw `as` casts on AU outside the newtype.
- For node-type capabilities, branch on `SwarmNodeType` from the api crate rather than duplicating the enum.

## Donts

- Do not import `libp2p` in `api`, `spec`, `forks`, `primitives`, `accounting/*`, or `localstore`. These are the libp2p-free layer.
- Do not use `underlay` in field names, error messages, or docs. The right word is `multiaddrs`.
- Do not add `serde_json` to a re-enabled crate without a workspace conversation. Several siblings are disabled for exactly that reason. The one settled exception is wire equivalence: a fixed-shape JSON object that must stay byte-identical to the live network (the swap cheque in `accounting/chequebook`) may use `serde_json` over a fixed-order wire struct. Tolerate JSON only for that, never for an internal or public API surface.
- Do not write architectural notes about the reference implementation inside crate-level rustdoc. If a comparison helps a reviewer, keep it short and factual, and never imply the reference is canonical.
- Do not write essay rustdoc. Module docs are a few terse lines (what the module is, plus the one load-bearing invariant), not multi-section essays; item docs only where the signature does not already say it; inline comments only where the code cannot speak for itself. See the Documentation section in `docs/agents/rust-idiomatic.md`. The builder, bandwidth, and rpc crates are the repeat offenders.
- Do not store `Arc<Identity>` clones in hot per-message paths. `Identity` is cheap to clone but the indirection compounds.

## Tests

- `cargo test -p vertex-swarm-<name>` per crate.
- `vertex-swarm-test-utils` provides `MockIdentity`, `MockTopology`, and cluster-shaped fixtures behind the `cluster` feature.
- `topology` and `node` rely on `libp2p-swarm-test` for behaviour tests; prefer that over hand-rolled mocks.
