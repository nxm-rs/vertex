# Redistribution primitives: benchmarks

Criterion benchmarks for the pure-compute redistribution primitives in
`vertex-swarm-redistribution` (`canonical_neighbourhood`, `reserve_sample`,
`make_inclusion_proofs`, and the per-chunk `AnyChunk::transformed_address` they
build on).

These measure speed only. Correctness (byte-exact conformance to the Swarm
storage-incentives protocol) is the job of the conformance suite,
`tests/conformance.rs`; keep both green.

## Running

```text
# protoc is needed to build the workspace.
nix-shell -p protobuf --run 'cargo bench -p vertex-swarm-redistribution'
```

Criterion writes HTML reports and `estimates.json` under `target/criterion/`.
Read the median estimate (`ns`) per benchmark id and the throughput line
(elements/s or bytes/s) Criterion prints.

## Benchmark groups

- `transformed_address/{cac,soc}` — the per-chunk anchor-keyed prefix-BMT
  consensus hash over a 4 KiB (`DEFAULT_BODY_SIZE`) body, for a content chunk and
  a single-owner chunk. Throughput is reported in **bytes/s** over the 4096-byte
  body, so it converts directly to MB/s. This is the heart of the sampler.
- `transformed_address/batch/{1000,10000}` — the aggregate hash-and-sort over a
  full candidate set: hash the transformed address of every chunk, then sort the
  set by transformed address. Throughput is reported in **bytes/s** over all
  hashed bodies, so the MB/s figure is directly comparable to another
  implementation's aggregate sample-hashing throughput at the same 1k / 10k
  sizes. Note this hashes and sorts the *whole* set; it is not the production
  `reserve_sample` min-16 selection (see below), which keeps only the smallest 16.
- `reserve_sample/{1024,10240,65536}` — the min-16 selection over a
  reserve-shaped candidate set. Each candidate carries a real typed CAC, so the
  per-chunk transformed-address work is realistic; this is the cost of the
  sorted-insertion selection over `N` chunks. Throughput is reported in
  elements/s. The transformed addresses are precomputed once at fixture-build
  time (via `SampleItem::new`, outside the measured region), so the group
  isolates the selection/insertion cost; the per-chunk hashing cost is the
  `transformed_address` group above.
- `make_inclusion_proofs/sample_16` — the proof-of-entitlement build over a full
  16-item sample (the RC chunk hash plus three witnesses, each an RC + OG + TR
  BMT proof). The sample is selected once outside the measured closure.
- `canonical_neighbourhood/{depth_0_all,depth_1_half}/N` for
  `N in {1024, 16384, 65536}` — the committed-depth membership filter.
  `depth_0_all` admits every address (pure filter cost); `depth_1_half` keeps
  roughly half the keyspace. Throughput is in elements/s.

Inputs are generated by a deterministic counter-based expander, not a random
source, so repeated runs use identical inputs and the numbers are reproducible.

## Reproducible measurement

Pin the CPU governor to `performance`, disable turbo, and run on an otherwise
idle host. Cross-machine numbers are not comparable. The figures reflect the
Rust BMT and Keccak implementations and codegen, not the algorithm, which is
fixed by the protocol.
