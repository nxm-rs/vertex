# Releasing vertex

Releases are cut locally by a maintainer with [`cargo-release`](https://github.com/crate-ci/cargo-release) and [`git-cliff`](https://github.com/orhun/git-cliff), then built by CI. The version bump, changelog, and signed tag are a deliberate local action; the binaries and Docker image are produced by GitHub Actions once the tag is pushed. Config lives in `release.toml` (cargo-release), `cliff.toml` (git-cliff), and `dist-workspace.toml` (cargo-dist).

The workspace is licensed `AGPL-3.0-or-later`. Vertex crates are not published to crates.io; the release artefacts are the prebuilt binaries and the container image.

## Versioning

Every crate shares a single workspace version, defined once in `[workspace.package].version` in the root `Cargo.toml`. They bump together (`shared-version = true`) and are tagged once per release as `vX.Y.Z`. The workspace is at `0.x`, so a `minor` bump is the normal release level and may carry breaking changes until `1.0`.

## What ships

`dist plan` builds the default bare `vertex` client for five targets:

- `x86_64-unknown-linux-gnu`
- `aarch64-unknown-linux-gnu`
- `x86_64-apple-darwin`
- `aarch64-apple-darwin`
- `x86_64-pc-windows-msvc`

Each archive carries the `vertex` binary plus `LICENSE` and `README.md`, alongside shell and PowerShell installers and SHA-256 checksums. The Docker workflow builds a multi-arch `linux/amd64` and `linux/arm64` image and pushes it to `ghcr.io/nxm-rs/vertex`.

## Prerequisites

- Your hardware signing key (YubiKey) unlocked in the gpg-agent, so cargo-release can create the signed release commit and signed `vX.Y.Z` tag without prompting.
- A clean checkout of `main` with no uncommitted changes, up to date with the remote.
- CI green on the commit you are about to release.
- `cargo-release` and `git-cliff` available. On the Nix dev shell: `nix-shell -p cargo-release git-cliff`.

## Cutting a release

From a clean `main`:

```bash
# Dry run first: shows the version bump, changelog, and tag.
cargo release minor

# When it looks right, execute it.
cargo release minor --execute
```

`cargo release minor --execute` will, in order:

1. Bump `[workspace.package].version` to the next minor.
2. Run the pre-release hook (`git-cliff`) to regenerate `CHANGELOG.md`.
3. Create the signed release commit `chore(release): X.Y.Z`.
4. Create the signed tag `vX.Y.Z`.
5. Push the release commit and tag to the remote.

Use `patch` instead of `minor` for a patch release, or `cargo release X.Y.Z --execute` to set an exact version.

## What the tag triggers

Pushing the `vX.Y.Z` tag starts two workflows:

- `.github/workflows/release.yml` (generated by cargo-dist): builds the five-target binary matrix, the installers, and the checksums, then creates the GitHub Release with the changelog body and uploads the artefacts.
- `.github/workflows/docker.yml`: builds and pushes the multi-arch image to `ghcr.io/nxm-rs/vertex`, tagged with the semver, the short sha, and `latest`.

Per-PR, both workflows run in a non-publishing mode: cargo-dist uploads throwaway build artefacts, and the Docker workflow builds `linux/amd64` only without pushing.

## Regenerating the release workflow

`.github/workflows/release.yml` is generated, not hand-authored. After editing `dist-workspace.toml`, regenerate it and commit the result:

```bash
nix-shell -p cargo-dist --run 'dist generate'
```

Keep `cargo-dist-version` in `dist-workspace.toml` pinned to the `dist` version you generate with.
