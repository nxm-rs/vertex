# Threat model: memory-bounded streaming chunk transfer (#289)

## Design (5-10 lines)

The core is two FFI-agnostic `Stream`s in `crates/swarm/node/src/stream.rs`: `get_stream(provider, addresses, config)` yields verified chunks for an ordered address list, and `put_stream(sender, chunks, config)` yields one receipt per uploaded chunk. Both build on the per-request outbound futures from #178 (each retrieval/push is a self-contained future correlated by its own substream) and hold them in a `FuturesOrdered`, so output order always matches input order regardless of completion order. The in-flight window is expressed in BYTES via `StreamConfig { window_bytes, max_concurrency }`: downloads reserve `MAX_CHUNK_BYTES` per in-flight retrieval (true size unknown until arrival), uploads reserve each chunk's real encoded size. A new request is admitted only when the reservation fits the window and the concurrency cap, and only when the stream is polled, so prefetch runs ahead of the consumer up to the window and stops the instant the consumer stops draining. Downloads deliver `VerifiedStampedChunk` only (#287): each chunk is proven to answer its requested address via `verify_answers` before it leaves the stream. The native FFI (`crates/ffi/src/api/client.rs`) pumps these streams into a flutter_rust_bridge `StreamSink<VertexChunkData>` / `StreamSink<VertexUploadAck>`; the wasm-bindgen adapter (`crates/swarm/node/src/stream_wasm.rs`) exposes the same core as JS async iterators. Both are thin adapters over the one core, which compiles native and wasm32.

## Trust boundary

The consumer (Dart/Swift/Kotlin host, or a browser via wasm-bindgen) is semi-trusted: it supplies address lists, chunk bytes, and a window budget, and it controls the drain rate. The remote peers serving retrievals and accepting pushes are untrusted and potentially adversarial. The boundary is the FFI/wasm bridge: strong domain types (`StampedChunk`, `VerifiedStampedChunk`, `ChunkAddress`) live only inside Rust; only flat bytes/primitives cross. Exactly one payload copy happens per direction at the bridge (`into_bytes().to_vec()` / `Uint8Array::from`); internally the chunk stays `Bytes`.

## Attacks, abuse, failure modes, and defenses

- Memory exhaustion by a long address list (host or attacker-controlled): the byte window caps outstanding payload. `get_stream` keeps at most `download_concurrency()` retrievals in flight; `put_stream` admits by real byte size against `window_bytes`. A million-address list never holds more than ~`window_bytes` in flight. Tested by `get_stream_caps_concurrency_at_the_byte_window` and `put_stream_caps_in_flight_bytes_at_the_window`.
- Slow-consumer memory growth (backpressure): the streams advance only when polled, and the adapters poll one item per `next()` / pump-loop iteration. A host that pauses its `StreamSink` listener or awaits the JS iterator slowly transitively pauses network reads. The `refill` happens on poll, so a drained slot admits exactly one replacement.
- Deadlock from a degenerate config: `StreamConfig::new` clamps `window_bytes` and `max_concurrency` to at least 1; a window smaller than one chunk still admits one request because the empty in-flight set bypasses the byte check. Tested by `config_clamps_zero_to_one`, `get_stream_window_smaller_than_chunk_still_streams_one_at_a_time`, and `put_stream_oversized_chunk_makes_progress_one_at_a_time`.
- Malicious peer returning wrong bytes for an address: `get_stream` calls `verify_answers`, which checks the chunk's content-derived address against the requested one. A mismatch becomes an `InvalidChunk` error item, never a chunk the consumer might trust. Tested by `get_stream_rejects_wrong_chunk_for_address`. The remote's error strings are never read (the underlying `ChunkTransferError::Remote` already strips them, per #296).
- Partial failure aborting the whole transfer: a per-address miss or per-chunk push failure surfaces as an `Err` item in that slot; the stream continues with the rest. Tested by `get_stream_surfaces_missing_chunk_as_error_item_and_continues` and `put_stream_surfaces_failed_push_and_continues`.
- Teardown leaks: when the host drops its `StreamSink` (closed sink), the FFI pump breaks its loop, dropping the core stream and all its in-flight futures, which cancels the outstanding network requests. Dropping the wasm iterator drops the boxed core stream the same way. Empty input terminates cleanly (`get_stream_empty_input_terminates`, `put_stream_empty_input_terminates`).
- Integer overflow on byte accounting: upload reservation release uses `saturating_sub`, so a logic slip cannot panic the pipeline. The FFI `u64` window is `try_from`-clamped to `usize::MAX` on a 32-bit host.

## Deferred gaps

- The wasm-bindgen adapter takes `Arc<dyn SwarmChunkProvider>` / `Arc<dyn SwarmChunkSender>` because the browser client builder is still native-only (`vertex-swarm-builder`, per docs/agents/wasm.md). Wiring the adapter to a real browser-built provider is deferred to the wasm client builder milestone tracked under the wasm cone work; the adapter and core are exercised through the native FFI path and core unit tests today.
- The download byte window reserves a fixed `MAX_CHUNK_BYTES` per slot rather than the chunk's true (often smaller) size, so the effective payload ceiling is conservative, never exceeded. A tighter post-arrival accounting is a possible future refinement but is not a safety gap.
- No on-the-wire protocol change is introduced, so no `SwarmHardfork` gate is required.
