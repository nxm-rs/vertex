# Threat model: memory-bounded streaming chunk transfer (#289)

## Design

The core is two FFI-agnostic `Stream`s in `crates/swarm/node/src/stream.rs`: `get_stream(provider, addresses, config)` yields verified chunks for an ordered address list, and `put_stream(sender, chunks, config)` (plus its fallible sibling `try_put_stream`) yields one receipt per uploaded chunk. Both build on the per-request outbound futures from #178 (each retrieval/push is a self-contained future correlated by its own substream) and hold them in a `FuturesOrdered`, so output order always matches input order regardless of completion order. The in-flight window is expressed in BYTES via `StreamConfig { window_bytes, max_concurrency }`: downloads reserve `MAX_CHUNK_BYTES` per in-flight retrieval (true size unknown until arrival), uploads reserve each chunk's real encoded size. A new request is admitted only when the reservation fits the window and the concurrency cap, and only when the stream is polled, so prefetch runs ahead of the consumer up to the window and stops the instant the consumer stops draining. Downloads deliver `VerifiedStampedChunk` only (#287): each chunk is proven to answer its requested address via `verify_answers` before it leaves the stream.

Both host bindings are pull-based, so the host's pull rate is the backpressure signal end to end. The native FFI (`crates/ffi/src/api/client.rs`) returns opaque handles `VertexDownloadStream` / `VertexUploadStream`; the host awaits their async `next()` once per item. The wasm-bindgen adapter (`crates/swarm/node/src/stream_wasm.rs`) returns `WasmGetStream` / `WasmPutStream`, JS async iterators whose `next()` polls the core once per `await`. Neither binding spawns a pump that pushes into a host queue: the core advances only when the host pulls, so a host that stops pulling pauses the core's polling and therefore the network reads. Both adapters are thin wrappers over the one core, which compiles native and wasm32.

## Trust boundary

The consumer (Dart/Swift/Kotlin host, or a browser via wasm-bindgen) is semi-trusted: it supplies address lists, chunk bytes, and a window budget, and it controls the pull rate. The remote peers serving retrievals and accepting pushes are untrusted and potentially adversarial. The boundary is the FFI/wasm bridge: strong domain types (`StampedChunk`, `VerifiedStampedChunk`, `ChunkAddress`) live only inside Rust; only flat bytes/primitives cross. Exactly one payload copy happens per direction at the bridge. Download-out copies once (`into_bytes().to_vec()` / `Uint8Array::from`). Upload-in copies once: the bridge copies the host bytes into the boundary `Vec<u8>`, and `reconstruct_upload` moves that `Vec` into `Bytes` via the zero-copy `Vec -> Bytes` conversion, so no second copy is made; internally the chunk stays `Bytes`.

## Backpressure: why a slow host pauses the network (the core property)

This is the load-bearing claim, and it is enforced by construction rather than by a sink.

- Both `get_stream` and `put_stream` only issue a new request inside their `refill`, which runs only when the stream is polled, and admit at most one replacement per drained slot. A host that does not poll issues no further requests.
- The FFI handles do not spawn a forwarding pump. Each `VertexDownloadStream::next` / `VertexUploadStream::next` polls the core exactly once and returns the single item to the host. flutter_rust_bridge turns each call into a Dart `Future` the host must await; until the host awaits the next one, the core is not polled, so no further retrievals or pushes start and nothing accumulates on the host's behalf. (This replaces the earlier design that pushed every completed item into flutter_rust_bridge's `StreamSink`, a non-blocking fire-and-forget post to the Dart port that applied no backpressure and could grow the Dart-side queue without bound for a long list against a slow listener. That sink is now used only by the low-volume logging surface, where unbounded buffering is acceptable.)
- The wasm adapter is identically pull-based: `WasmGetStream::next` / `WasmPutStream::next` are `async fn`s the JS host awaits, polling the core once per await.

Upload memory is bounded the same way the network window is: `put_stream` pulls its `StampedChunk`s lazily from a `Peekable` source iterator and materializes a chunk only when it admits the push, so at most a window's worth of chunks are reconstructed at once. The FFI feeds a reconstruct-on-demand iterator, so the only chunks built into strong types at any moment are the in-flight ones. The host still owns the `Vec<VertexChunkUpload>` it passed by value (an unavoidable cost of a by-value API), but Rust holds no second resident copy of that list and copies no chunk payload a second time.

## Attacks, abuse, failure modes, and defenses

- Memory exhaustion by a long address/chunk list (host or attacker-controlled): the byte window caps outstanding payload. `get_stream` keeps at most `download_concurrency()` retrievals in flight; `put_stream` admits by real byte size against `window_bytes` and materializes chunks lazily. A million-entry list never holds more than ~`window_bytes` in flight. Tested by `get_stream_caps_concurrency_at_the_byte_window`, `put_stream_caps_in_flight_bytes_at_the_window`, and `put_stream_materializes_chunks_lazily_within_the_window` (a 1024-item lazy feed against a gated sender materializes at most window+1 chunks).
- Slow-consumer memory growth (backpressure): covered above. The core advances only when polled, and both bindings poll exactly one item per host pull, so a paused host pauses the network reads with nothing buffered past the window on the host's behalf. There is no spawned pump and no host-side push queue.
- Deadlock from a degenerate config: `StreamConfig::new` clamps `window_bytes` and `max_concurrency` to at least 1; a window smaller than one chunk still admits one request because the empty in-flight set bypasses the byte check. Tested by `config_clamps_zero_to_one`, `get_stream_window_smaller_than_chunk_still_streams_one_at_a_time`, and `put_stream_oversized_chunk_makes_progress_one_at_a_time`.
- Malicious peer returning wrong bytes for an address: `get_stream` calls `verify_answers`, which checks the chunk's content-derived address against the requested one. A mismatch becomes an `InvalidChunk` error item, never a chunk the consumer might trust. Tested by `get_stream_rejects_wrong_chunk_for_address`. The remote's error strings are never read (the underlying `ChunkTransferError::Remote` already strips them, per #296).
- Partial failure aborting the whole transfer: a per-address miss or per-chunk push failure surfaces as an `Err` item in that slot; the stream continues with the rest. A malformed upload chunk (byte/address mismatch, bad stamp) is fed through `try_put_stream` as an error in that chunk's slot rather than aborting the whole upload, while a malformed address is still rejected up front. Tested by `get_stream_surfaces_missing_chunk_as_error_item_and_continues`, `put_stream_surfaces_failed_push_and_continues`, and `try_put_stream_surfaces_feed_error_in_order_and_continues`.
- Teardown leaks: the host owns the handle. Dropping `VertexDownloadStream` / `VertexUploadStream` (or the wasm iterator) drops the core stream and all its in-flight futures, which cancels the outstanding network requests. Empty input terminates cleanly (`get_stream_empty_input_terminates`, `put_stream_empty_input_terminates`).
- Integer overflow on byte accounting: upload reservation release uses `saturating_sub`, so a logic slip cannot panic the pipeline. The FFI `u64` window is `try_from`-clamped to `usize::MAX` on a 32-bit host.

## Known gaps and deferrals

- Head-of-line liveness, not memory (low, deferred): `get_stream` yields strictly in input order via `FuturesOrdered`, and the retrieval protocol carries no per-request deadline. A peer that accepts the substream for `address[0]` and never delivers holds the head slot open; later addresses complete but cannot be yielded, so the transfer stalls with no progress and no error. Memory stays bounded (the stalled future still counts against the window, so refill stops), so this is availability, not exhaustion, and the same no-deadline exposure already exists for the single-chunk `download_chunk`. The fix is a per-retrieval deadline in the retrieval protocol (a `SwarmError` item on timeout so the stream advances), which is a protocol-layer change tracked separately rather than bolted onto the streaming adapter; it is not introduced here. The streaming pipeline requires the underlying provider to enforce its own retrieval deadline.
- The wasm-bindgen adapter takes `Arc<dyn SwarmChunkProvider>` / `Arc<dyn SwarmChunkSender>` and its constructors are plain `pub fn`, because the browser client builder is still native-only (`vertex-swarm-builder`, per docs/agents/wasm.md). The constructors are not yet `#[wasm_bindgen]`-exported and JS cannot build a provider to pass them today; wiring a JS-constructible provider facade is deferred to the wasm client builder milestone tracked under the wasm cone work. The adapter compiles on wasm32 (CI-enforced) and shares the exact core path the native FFI exercises; the core stream behavior (ordering, window, lazy upload, verify-before-yield, inline errors) is covered by the native unit tests above.
- The download byte window reserves a fixed `MAX_CHUNK_BYTES` per slot rather than the chunk's true (often smaller) size, so the effective payload ceiling is conservative, never exceeded. A tighter post-arrival accounting is a possible future refinement, not a safety gap.
- No on-the-wire protocol change is introduced, so no `SwarmHardfork` gate is required.
